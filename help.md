# Atomic Choice — руководство по запуску и использованию

Платформа анонимного голосования для МИФИ:

- FastAPI-сервер раздаёт SPA `static/atomic_choice.html` и API.
- Локальная сеть Hardhat хранит контракты `Whitelist`, `VotingFactory`, `VotingPoll`.
- Авторизация: вход по приватному ключу, проверка по on-chain вайтлисту.
- Анонимность: nullifier hash + ZK-stub + сокрытие результатов до окончания.

## 2. Установка зависимостей (5–10 минут)

```bash
# 1. Перейдите в папку проекта
cd atomic-choice

# 2. Python окружение
python -m venv .venv
source .venv/Scripts/activate   # Windows / Git Bash
# или: .venv\Scripts\activate   # Windows / PowerShell
# или: source .venv/bin/activate  # macOS / Linux
pip install -r requirements.txt

# 3. Hardhat и контракты
cd contracts
npm install
npx hardhat compile             # создаёт contracts/artifacts/...
cd ..
```

После `compile` появится папка `contracts/artifacts` — оттуда берётся
байт-код для деплоя.

## 3. Первый запуск

Открой **3 терминала** в корне `atomic-choice/`.

### Терминал 1 — Hardhat node

```bash
cd contracts
npx hardhat node
```

Терминал начнёт логировать «`Started HTTP and WebSocket JSON-RPC server at http://127.0.0.1:8545/`».
Не закрывай его до конца сессии.

### Терминал 2 — FastAPI

```bash
# из корня проекта, с активной venv
uvicorn main:app --reload --port 8000
```

При первом старте `main.py`:

1. Сгенерирует **10 keypair** (приватный + публичный ключ) → `keys.txt` и `keys.json`.
2. Создаст серверный `server_salt.bin` (нужен для анонимности seed-участников).
3. Если `deployments.json` уже есть — подтянет адреса контрактов и
   синхронизирует Merkle-дерево.

### Терминал 3 — деплой и сид-данные

```bash
curl -X POST http://localhost:8000/api/admin/setup
```

Этот вызов делает всё за один клик:

- деплоит `PoseidonStub`, `VerifierStub`, `Whitelist`, `VotingFactory`;
- регистрирует **seed-студентов** и **10 keypair-участников** в глобальном
  вайтлисте (отправляет `addCommitment` на чейн);
- создаёт **3 демо-голосования**.

После этого:

- адреса контрактов сохранены в `deployments.json`;
- приватные ключи участников лежат в `keys.txt` (по одной строке на участника);
- сервер готов принимать запросы.

## 4. Вход в систему

Открой <http://localhost:8000>. Вы увидите модалку входа.

1. Откройте `keys.txt` в корне проекта.
2. Скопируйте строку `Приватный ключ: 0x…` любого участника.
3. Вставьте её в поле «приватный ключ» и нажмите **Войти**.

Что происходит под капотом:

1. Браузер локально (через `ethers.js`) выводит публичный адрес из ключа.
   **Сам ключ никогда не покидает браузер.**
2. Браузер запрашивает у сервера `/api/students/{адрес}` — нет ли его в
   реестре и стоит ли флаг `whitelisted`.
3. Если оба условия выполнены — сессия записана в `sessionStorage` и
   модалка закрывается.

Имя/адрес участника появятся в правом верхнем углу. Кнопка «Выйти»
очищает сессию.

> Если адрес есть в `keys.txt`, но логин не работает с ошибкой «не в
> вайтлисте» — значит вы запустили `setup` до того, как `keys.txt` был
> сгенерирован. Запросите `/api/admin/whitelist/keypairs`, чтобы добавить
> ключи в вайтлист.

## 5. Главные сценарии в UI

### 5.1 Просмотр голосований

- **Главная** — активные голосования карточками.
- **Голосования** — все, с фильтром (Все / Активные / Запланированные / Завершённые).
- **Мои опросы** — голосования, в которых текущий пользователь является создателем.

### 5.2 Создание голосования

1. Нажмите «Создать» в навигации.
2. Заполните: тема, название (≥3 символов), описание, начало, конец, варианты ответов (≥2).
3. «Сохранить голосование» → `POST /api/admin/polls` с `creator_wallet = ваш адрес`.
4. После успеха попадаете на вкладку «Мои опросы».

Поведение в соответствии с Use Case:

- название обязательно;
- минимум 2 варианта ответа;
- дата окончания > дата начала;
- дата начала ≥ текущего момента (с допуском 60 секунд);
- расширения 5а/5б/5в реализованы тостами с понятными ошибками.

### 5.3 Управление вайтлистом своего голосования

В деталях голосования (если вы его создатель) появляется блок «Вайтлист
голосования». Вставьте адрес участника и нажмите «Добавить» — он сможет:

- войти на сайт по своему приватному ключу;
- голосовать **только в этом опросе** (в остальные не пускает ни UI, ни сервер).

Если адрес вообще не был зарегистрирован, он автоматически создаётся как
«гостевой» участник с детерминированным ZK-секретом из server salt + wallet.
На контракт ушлёт `addWhitelistRoot(newRoot)`. Список участников вайтлистов
сохраняется в `poll_whitelists.json` и переживает рестарт сервера.

### 5.4 Голосование

1. Откройте активное голосование, выберите вариант, «Проголосовать».
2. **Браузер подписывает голос приватным ключом** (EIP-191 personal_sign).
   Подписывается каноническое сообщение вида:
   ```
   Atomic Choice — confirm vote
   poll: 0x…
   option: <idx>
   nonce: <random>
   ```
3. Сервер `Account.recover_message(...)` восстанавливает адрес → подтверждает
   владение PK → проверяет per-poll/глобальный вайтлист.
4. Сервер строит nullifier `Poseidon(secret, pollId)` и зарелеит
   `castVote()` от имени deployer (msg.sender анонимен on-chain).
5. Контракт проверяет nullifier на повтор и принимает голос.
6. UI показывает «Голос подписан и анонимно учтён ✓».

Повторное голосование заблокировано:
- контрактом (`nullifierUsed[hash] = true`);
- сервером (антиреплей по `(poll, wallet, nonce)`);
- проверкой подписи (нельзя проголосовать чужим адресом).

### 5.5 Результаты

- Активные голосования: результаты скрыты для всех, кроме создателя
  и того, кто уже проголосовал.
- Завершённые: результаты доступны всем.
- API `/api/polls/{addr}/results` возвращает 403 пока голосование активно.

## 6. Анонимность — что гарантируется

| Уровень | Видит ли личность? |
|---|---|
| Любой внешний наблюдатель (читает контракт) | нет — `VoteCast` содержит только `(nullifierHash, option)` |
| Создатель голосования через UI/API | нет — список «кто голосовал» не выдаётся ни в каком эндпоинте |
| Другие участники | нет — то же |
| Администратор сервера | да (по логам) — оставлено для anti-fraud в соответствии с FR 2/7 |

Дополнительно:

- on-chain транзакция всегда подписана адресом deployer (не личным).
- nullifier seed-участника зависит от серверного `server_salt.bin`,
  поэтому угадать его из публичного адреса невозможно.
- результаты до окончания голосования замаскированы нулями в API
  (флаг `results_hidden`).

## 7. Полезные эндпоинты

Все эндпоинты — JSON. Документация Swagger: <http://localhost:8000/docs>.

### Админ

| Метод | URL | Что делает |
|---|---|---|
| GET  | `/api/admin/status`               | состояние ноды, контрактов, ключей |
| POST | `/api/admin/setup`                | полная инициализация (deploy + whitelist + seed polls) |
| POST | `/api/admin/deploy`               | только деплой контрактов |
| GET  | `/api/admin/whitelist`            | глобальный root + size |
| POST | `/api/admin/whitelist/{wallet}`   | добавить адрес в глобальный вайтлист |
| POST | `/api/admin/whitelist/keypairs`   | вайтлистить все keypair из `keys.txt` |
| POST | `/api/admin/sync`                 | пересобрать Merkle-дерево из событий |
| POST | `/api/admin/polls`                | создать голосование (`creator_wallet` опционально) |
| GET  | `/api/admin/keypairs`             | публичные адреса (без приватных) |

### Пользовательские

| Метод | URL | Что делает |
|---|---|---|
| GET  | `/api/students`               | весь реестр участников |
| GET  | `/api/students/{wallet}`      | один участник (используется при логине) |
| GET  | `/api/polls`                  | список голосований (без приватных результатов) |
| GET  | `/api/polls/{addr}`           | детали голосования |
| GET  | `/api/polls/{addr}/results`   | результаты (только когда status=ended) |
| GET  | `/api/polls/{addr}/whitelist` | вайтлист этого голосования |
| POST | `/api/polls/{addr}/whitelist` | добавить участника (только создатель) |
| POST | `/api/vote`                   | голосование |
| GET  | `/api/merkle-proof/{wallet}`  | глобальный Merkle proof |

## 8. Перезапуск и сброс состояния

Если вы хотите начать с нуля:

```bash
# остановите оба процесса (Hardhat и uvicorn)
rm -f deployments.json keys.txt keys.json server_salt.bin poll_whitelists.json
# и заново запустите Hardhat + uvicorn + curl /api/admin/setup
```

Удалив только `deployments.json` (и оставив `keys.txt`), вы переразвернёте
контракты, но keypair-участники останутся теми же. Если меняете состав
seed-студентов — также сотрите `server_salt.bin`, чтобы синхронизировать
их commitments.

## 9. Отладка частых проблем

| Симптом | Причина | Решение |
|---|---|---|
| «Hardhat node not connected» | не запущен `npx hardhat node` | запустите его в отдельном терминале |
| «WHITELIST_ADDRESS not set» | контракты не задеплоены | `POST /api/admin/setup` |
| Логин падает «не в вайтлисте» | keypairs не добавлены в whitelist | `POST /api/admin/whitelist/keypairs` |
| «Вы уже проголосовали» | nullifier уже использован | это и есть защита от двойного голоса |
| «Merkle root не принят контрактом» | дерево разошлось с on-chain | `POST /api/admin/sync` |
| Артефактов нет, deploy падает | не запускали `npx hardhat compile` | выполните внутри `contracts/` |

## 10. Соответствие требованиям

- **FR 1 (авторизация)** — вход по приватному ключу с проверкой адреса по on-chain Merkle whitelist (`Whitelist.sol` + off-chain дерево).
- **FR 2 (анонимность)** — msg.sender deployer, nullifier hash, скрытие результатов до окончания, серверный salt для seed-секретов.
- **FR 3 (уникальность)** — `nullifierUsed` mapping в `VotingPoll.sol`.
- **FR 4 (создание)** — UI `Создать` + `POST /api/admin/polls`, валидация по Use Case.
- **FR 5 (доступ к голосованию)** — глобальный + per-poll вайтлист, проверки в `cast_vote`.
- **FR 6 (верифицируемость)** — все смарт-контракты публичные, события on-chain (`VoteCast`, `PollEnded`, `CommitmentAdded`).
- **FR 7 (защита)** — nullifier на чейне, deployer-only-deploys, проверка прав создателя на изменение whitelist.

User Stories 1–5 покрыты UI-сценариями выше.
